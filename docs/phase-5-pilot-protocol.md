# Phase 5 Small Pilot Protocol

## Participants and privacy

Recruit 5–10 beginner or early Python learners. Explain that the activity records a random participant code, timestamps, interaction events, answer correctness, and source hashes. It records no name, email, or source text in the event stream. Obtain consent before opening a lesson.

## Session: 35–50 minutes

1. Introduce the participant code, local-only storage, export, and deletion controls.
2. Ask the learner to complete the pre-check without assistance.
3. Complete one or two lessons using prediction, reveal, guided Lens inspection, variation, recognition, and Build.
4. Let the learner submit the post-transfer task without copying the lesson example.
5. Conduct a five-minute interview:
   - Explain the code's structure in your own words.
   - Which Lens view helped, and which was confusing?
   - What changed between the two scenarios?
   - Where did you feel overloaded or need facilitator help?

## Facilitation rules

- Do not reveal answers before the learner commits.
- Record facilitator rescue separately in session notes; do not edit the learner's source.
- Treat failed Build criteria as instructional evidence, not participant failure.
- If the interface blocks progress, record the confusion before helping.
- Use **Restart lesson** for a new attempt stream. Use **Delete pilot data** when consent is withdrawn.

## Export and interpretation

Use **Export JSON** at the end of the session. Confirm the file contains the participant code, attempts, events, summaries, and lesson versions. Store exported files according to the study's consent and retention policy.

The proof is promising when most learners finish without rescue, post-transfer performance exceeds pre-check performance, unchanged Build never passes, and no recurring interface problem affects most participants. Completion alone must not be reported as learning effectiveness.
